# Skills Provenance

🌐 日本語: [skills-provenance.ja.md](skills-provenance.ja.md)

← [Docs index](../README.md)

Every skill under `~/.agents/skills/` belongs to exactly one of five provenance categories.
The classification is based on **management method**, not authorship — an official Anthropic skill declared in `.chezmoiexternal.toml` is `external`, not `system`.

---

## The five categories

| Category | Definition | Physical location |
|---|---|---|
| `curated` | chezmoi SSOT — authored or forked, checked into this repo | `home/dot_agents/skills/<name>/` → `~/.agents/skills/<name>/` |
| `external` | Declared in `.chezmoiexternal.toml`, fetched from GitHub tarballs with a pinned SHA | `~/.agents/skills/<name>/` (not in chezmoi source) |
| `system` | Anthropic-distributed system skills; not managed by this repo | `~/.agents/skills/.system/<name>/` |
| `evolved` | Generated by the CLV2 `/evolve` flow from reviewed instinct clusters | `$CLV2_HOMUNCULUS_DIR/evolved/skills/<name>/` — **outside** `~/.agents/skills/` |
| `unmanaged` | Does not fit any of the above — **policy violation** | — |

`evolved` skills are intentionally placed outside `~/.agents/skills/` and are not symlinked into the standard skill discovery path. They exist in the homunculus dir, managed by the CLV2 continuous-learning system.

`unmanaged` skills are detected by `tests/skill_provenance.bats` (informational runtime check) and must be either deleted or absorbed into `curated` or `external`.

## Adding a curated skill

1. Create the skill directory in the chezmoi source:

   ```
   home/dot_agents/skills/<name>/
   ```

2. Add at least a `SKILL.md` (and any scripts your skill needs). The source is deployed to `~/.agents/skills/<name>/` by `chezmoi apply`.

3. Verify the skill is not also declared `external` — the no-overlap rule is enforced by `tests/skill_provenance.bats`.

---

## Adding an external skill (ECC / Anthropic)

External skills are not checked into the repo source. They are fetched from GitHub archive tarballs by chezmoi at apply time.

**For ECC skills**: append the skill name to the `[ecc].skills` array in `home/.chezmoidata.toml`:

```toml
[ecc]
  commit = "<pinned-commit-sha>"
  skills = [
    # ... existing entries ...
    "your-new-skill",
  ]
```

The `home/.chezmoiexternal.toml` file contains a `{{ range $skill := .ecc.skills }}` block that emits one external entry per element, all version-locked to `[ecc].commit`. A single ECC tarball is downloaded and cached by chezmoi; adding an entry costs no additional download.

**For Anthropic system skills**: add a literal `[".agents/skills/<name>"]` entry to `home/.chezmoiexternal.toml`, following the `stripComponents = 3` pattern already used for the 17 existing Anthropic entries.

### The never-both rule

A skill name must never appear in **both** `home/dot_agents/skills/<name>/` (curated) and in the external declarations. chezmoi would attempt to deploy a directory from source and fetch an external to the same path, which produces a conflict. The provenance bats test asserts this for all curated skills against both the literal external headers and every element of `[ecc].skills`.

## The `agent-browser` discovery-stub pattern

`agent-browser` is curated as a **discovery stub only** (`home/dot_agents/skills/agent-browser/SKILL.md`). Its specialized skills (electron, slack, dogfood) are **not vendored** in this repo. They would go stale quickly and are served at runtime by the agent-browser CLI itself:

```
agent-browser skills get <name>
```

The previously-deployed copies of the specialized skills are force-removed via `home/.chezmoiremove`:

```
.agents/skills/electron
.agents/skills/slack
.agents/skills/dogfood
```

Removing a file from the chezmoi source does not delete an already-deployed copy; `.chezmoiremove` is what tells chezmoi to ensure those paths are absent on every apply.

## Machine enforcement: `tests/skill_provenance.bats`

`tests/skill_provenance.bats` is the CI gate for the provenance policy. It runs in every CI pipeline (bats-only job, no chezmoi required). Its assertions fall into two tiers:

**Deterministic source-side assertions** (always fail CI on violation):

- `agent-browser` vendors only its discovery stub; `electron`/`slack`/`dogfood` are absent from source and listed in `.chezmoiremove`.
- Every curated skill directory in source is non-empty (contains at least one regular file at depth ≤ 2).
- `.chezmoiexternal.toml` declares at least one `[".agents/skills/..."]` external entry, and includes the ECC range block wired to `[ecc].commit`.
- No skill name appears in both the curated source tree and the external declarations (never-both rule).
- `home/AGENTS.md.tmpl` documents all five provenance categories.
- Retired unmanaged skills (`agentcore`, `vercel-sandbox`, `patch-remote-control`, `find-skills`) are absent from source.
- Orphaned `sdd-*` subagents are absent from `home/dot_claude/agents/`.
- The ECC skills range block in `.chezmoiexternal.toml` is wired correctly (url, include glob, `stripComponents = 3`), contains ≥ 100 entries, and has no duplicates.
- No ECC skill name collides with a curated skill of the same name.

**Informational runtime check** (always returns exit 0):

- Scans `~/.agents/skills/` for skills that are neither curated (in chezmoi source), nor external (declared in `.chezmoiexternal.toml` or in `[ecc].skills`), nor under `.system/`. Reports any found as `unmanaged` to the bats tap output but does not fail CI.

### How the test resolves ECC skill names without chezmoi

The `[ecc].skills` array in `.chezmoidata.toml` is the source of truth for the adopted ECC skill names (authoritative count: see the External skill inventory table above). The range block in `.chezmoiexternal.toml` only contains the literal template variable `{{ $skill }}` — a plain `grep` of the external file cannot see the expanded names.

The test uses a chezmoi-free `awk` scope to extract the list directly from `.chezmoidata.toml`:

```bash
awk '
  /^\[ecc\]$/        { in_ecc = 1; next }
  /^\[/              { in_ecc = 0; in_list = 0 }
  in_ecc && /^[[:space:]]*skills[[:space:]]*=[[:space:]]*\[/ { in_list = 1; next }
  in_ecc && in_list && /^[[:space:]]*\]/ { in_list = 0; next }
  in_ecc && in_list  { print }
' "${HOME_DIR}/.chezmoidata.toml" | grep -oE '"[^"]+"' | tr -d '"'
```

The scope (`in_ecc && in_list`) ensures that an unrelated `skills` key added to a different TOML section cannot perturb the result.
