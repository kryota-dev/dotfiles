# セッション transcript の汚染コンテキスト駆除

## これは何か

Claude Code のセッション履歴 JSONL を、「ここまで残したい」メッセージ（leaf）で切り詰め、
それ以降の汚染された会話（誤ったツール出力に基づく調査・思考、無関係な脱線、中断など）を削除する。
leaf はユーザーが起点を明示すればそれを使い、明示がなければ**エージェントが会話を読んで汚染起点を判断**する。
目的は **セッションを作り直さず、汚染前の地点から `--resume` で再開できる状態に戻すこと**。

JSONL は「1行 = 1イベント」の追記ログで、`parentUuid` でメッセージツリーを成す。末尾を削るだけなら
ツリーの整合性は自動的に保たれるが、**resume の分岐点は `last-prompt` エントリの `leafUuid` が決める**。
ここを取りこぼすと、再開時に意図より手前のターンへ戻ってしまう。本スキルはこの落とし穴を含めて機械的に処理する。

## 安全原則（必ず守る）

- **実行中の自セッションは編集しない。** `${CLAUDE_SESSION_ID}` を必ずスクリプトへ渡し、対象がそれと
  一致したら拒否させる。汚染されたのは通常「別セッション」であり、ユーザーは今このセッションから依頼している。
- **書き込み前に必ず dry-run でプランを提示し、ユーザーの明示的な確認を得てから `--apply` する。** 破壊的操作。
- **必ずバックアップを取る**（スクリプトが `.bak-<timestamp>` を自動生成）。ユーザーが作成した履歴であり、
  万一に備えて完全復元できる状態を残す。
- 保持行は再シリアライズせず原文のまま書き出す（既存行へのフォーマット差分混入を防ぐ）。
- 保持行が全て有効 JSON であることを検証し、不正があれば書き込まない（スクリプトが担保）。

## 手順

スクリプトは `${CLAUDE_SKILL_DIR}/scripts/prune_transcript.py`（Python3・標準ライブラリのみ）。

### 1. 対象 JSONL を特定する

ユーザーがフルパスを提示していればそれを使う。曖昧なら一覧から特定する:

```bash
python3 "${CLAUDE_SKILL_DIR}/scripts/prune_transcript.py" list --limit 20
```

`~/.claude*/projects/`（`~/.claude` と `~/.claude-r06` 等の派生環境を一括）と `$CLAUDE_CONFIG_DIR/projects/`
を更新時刻順に表示する。ファイル名（拡張子除く）がセッションIDで、resume 時に指定する値でもある。

### 2. 切り詰め点（leaf = 残す最後のメッセージ）を特定する

まず会話の骨格を把握する。先頭 `*` は resume 安全な leaf 候補（Claude Code がターン境界として
記録した点。構造的事実であり内容判定ではない）:

```bash
python3 "${CLAUDE_SKILL_DIR}/scripts/prune_transcript.py" map <jsonl>
```

#### (A) ユーザーが起点を明示している場合

「○○と回答した以降を破棄」のように起点が示されていれば、map でその発言（`AskUserQuestion` 回答や
ユーザー発言）を見つけ、それを leaf にする。汚染は通常その**直後の assistant ターン**から始まるため、
leaf 自体は汚染前として残す。

#### (B) 起点が明示されていない場合 — 文脈読解で汚染起点を特定する

**キーワードや定型パターンに頼らず、会話の中身を読んで「どこから内容が壊れ始めたか」を自分で判断する。**
判定スクリプトは持たない。これは語彙や状況に依存しない、再現性の高いやり方。

1. map で全体の流れを掴み、怪しい領域を絞る。
2. その領域の本文を `show` でデコード表示して精読する（生 JSON より低コスト）:

   ```bash
   python3 "${CLAUDE_SKILL_DIR}/scripts/prune_transcript.py" show <jsonl> --range 95-130
   ```

   `【TEXT】【THINK】【TOOL_USE】【TOOL_RESULT】` のブロック単位で読める。
3. 「汚染が始まった最初のターン」を意味で見極める。判断の手掛かり（**例示であり判定条件ではない**）:
   - ツール結果に、実行コマンドと無関係な文字列・ノイズ・指示文らしき断片が紛れている
   - その後の assistant の推論・記述が、その紛れ込んだ内容に引きずられている／前提が崩れている
   - 同じ調査・修正を不自然に繰り返している、堂々巡りになっている
   - assistant 自身が出力の異常・不整合に言及している
   - ユーザーが困惑・指摘・中断している
   どれも「ヒント」であって、最終判断は前後関係の読解で行う。1つの語に飛びつかない。
4. 汚染が始まった最初のターンを決めたら、その**直前のターン境界**（map の `*`）を leaf に選ぶ。
   - **汚染域内の `*` を選ばない**こと。`*` は単に「ターン境界」であり「クリーン」を意味しない。
   - leaf を仮決めしたら、`show` で「leaf 自体はクリーンか」「leaf より後ろは全て破棄してよい汚染か」を
     最終確認する。汚染がさらに手前から始まっていれば leaf をより前へ動かす。

### 3. dry-run でプランを提示し、確認を得る

leaf は行番号・uuid・含有文字列のいずれかで指定する（文字列指定は一意であること）:

```bash
python3 "${CLAUDE_SKILL_DIR}/scripts/prune_transcript.py" prune <jsonl> \
  --line <leafの行番号> \
  --current-session-id "${CLAUDE_SESSION_ID}"
```

出力される「切り詰めプラン」で次を確認しユーザーに提示する:
- leaf メッセージの内容・行・uuid
- 保持境界（leaf 直後の `last-prompt` 等メタ行を取り込んだ行）と 保持/削除 行数
- **resume 整合性**: `OK` なら保持領域の最後の `last-prompt.leafUuid` が leaf と一致＝回答直後から再開可能。
  `要注意` の場合は §5 を参照。

**ここで必ずユーザーの確認を取る**（どの発言まで残すか、削除範囲が想定通りか）。

### 4. 確認後に適用する

```bash
python3 "${CLAUDE_SKILL_DIR}/scripts/prune_transcript.py" prune <jsonl> \
  --line <leafの行番号> --apply \
  --current-session-id "${CLAUDE_SESSION_ID}"
```

バックアップ作成 → 保持行の JSON 検証 → atomic 置換 → 検証、までを行い結果を表示する。
結果（保持行数・バックアップパス・resume 整合性）をユーザーに報告する。

### 5. resume 整合性が「要注意」のとき

leaf 直後にその leaf を指す `last-prompt` が無い（＝ターンの途中で切った）場合、resume が手前のターンから
再開する恐れがある。次のいずれかで対処する:

- **推奨**: ターン境界のメッセージを leaf に選び直す（最も安全で副作用がない）。
- やむを得ない場合は `--fix-resume` を併用し、leaf を指す `last-prompt` をスクリプトに補正追記させる:

  ```bash
  python3 "${CLAUDE_SKILL_DIR}/scripts/prune_transcript.py" prune <jsonl> \
    --line <leafの行番号> --apply --fix-resume \
    --current-session-id "${CLAUDE_SESSION_ID}"
  ```

## 完了後の案内

- 対象セッションを `--resume`（または該当 UI から再開）すると、leaf の直後＝汚染前から続けられることを伝える。
- 切り詰め後の leaf が `AskUserQuestion` 回答などの tool_result の場合、再開後にユーザーが新しい指示を
  打てば、その回答を踏まえた状態から汚染なしで応答が再開する旨を補足する。
- 問題なければバックアップ（`.bak-<timestamp>`）は削除してよいこと、不満があれば `cp` で完全復元できることを伝える。

## 注意

- 対象が現在ブラウザ/別ウィンドウで開いている可能性があるときは、編集前に閉じてもらう。アクティブな
  セッションファイルを書き換えると、そのセッション側の追記と競合しうる。
- このスキルは Claude Code 内部の履歴ファイルを編集する。スクリプトは原文保持・バックアップ・検証で
  保守的に作ってあるが、Claude Code 本体のバージョン更新で JSONL スキーマが変わった場合は `map` 出力を
  確認し、メタ行種別（`META_TYPES`）の想定が現行と一致しているか点検する。
