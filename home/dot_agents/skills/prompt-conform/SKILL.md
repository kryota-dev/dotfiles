# prompt-conform

プロンプトを、**対象モデルの現行公式ベストプラクティスに準拠**する形へ書き換える。

## 中核原則: ベストプラクティスは毎回ライブで引く

プロンプトエンジニアリングの定石は**モデルの世代交代で変わる**。旧世代で有効だった書き方が
新世代では逆効果になることがある（過剰トリガー、非対応 API 等）。したがって:

- **この skill にベストプラクティスの中身をハードコードしない。** 陳腐化するため。
- 実行のたびに**対象モデル系の公式ドキュメントをライブ取得**し、そこに書かれている
  現行の指針だけを根拠にする。
- **自分の記憶（訓練データ）で代替しない。** 訓練データは古い可能性がある。必ず取得した
  ドキュメントの記述を一次ソースとして扱う。

`references/doc-sources.md` はドキュメントの**在り処（URL とその探し方）**のみを持つ。
定石そのものは持たない。

## ワークフロー

### Step 1: 対象モデル系を特定する

最適化対象のプロンプトが**どのモデルで実行される想定か**を決める。ここでドキュメントの
取得先が決まる。

1. **ユーザーが対象モデルを明示している場合**（例: 「GPT-5 向けに」「Opus 4.8 で使う」）→ それを採用。
2. 明示がなければ**現行セッションのモデル/harness を自動判定**する。Claude Code なら `claude-*`、
   Codex なら `gpt-*`/`codex-*` を手がかりに**モデル系（family）**を推定する。
   - ただし世代・マイナー版（例: `opus-4-8`）までの自己認識は不確実なことがある。版まで確信が
     持てなければ、family 単位の汎用ガイドに倒すか、ユーザーに確認する（誤った版のページに
     依拠するより安全）。
3. **対象が複数 harness / 複数 family 向け**（skill / AGENTS.md / CLAUDE.md など Claude と Codex の
   両方で使う文面）の場合は、現行セッションだけに寄せず**該当する全 family の公式ガイドを取得**し、
   共通化できる原則と family 固有の差分を分けて提示する。
4. どうしても判定できない場合はユーザーに確認する。

モデル系 → ドキュメント取得先の対応は `references/doc-sources.md` を参照。

### Step 2: プロンプトの種別と意図を抽出する

- **種別**を判定する: skill / slash command / タスク指示 / CLAUDE.md 節 / コード生成指示 /
  一般の依頼文。種別によって適用すべき定石（例: agentic 系、出力フォーマット制御、tool 起動）が変わる。
- 元プロンプトの**目的・制約・成功条件**を読み取る。
- **少しでも意図が曖昧なら、推測で埋めずユーザーに確認する。** 目的を誤読したまま最適化すると、
  文面は綺麗でも的外れなプロンプトになる。曖昧さの解消を最適化より優先する。

### Step 3: 公式ドキュメントをライブ取得する

`references/doc-sources.md` の URL を起点に、対象モデル系の**現行**プロンプトガイドを取得する。

- WebFetch / web 取得ツールを使い、実際にページを読む。**記憶で済ませない。**
- リダイレクトが返ったら追う（例: `docs.anthropic.com` → `platform.claude.com`）。
- 索引ページに**モデル別ページ**へのリンクがあれば、対象モデルに一致するものを辿る
  （例: Claude の best-practices 索引 → `prompting-claude-opus-4-8`）。
- ページが大きく一度に読めない場合は、対象プロンプトに関係するセクションに絞って読む。

### Step 4: 元プロンプトを取得ガイドと突き合わせる

取得した現行ガイドに照らして、元プロンプトのギャップを洗い出す。

- **適用する観点は毎回、取得したガイドが強調している項目から抽出する。** この skill 側に定石
  チェックリストを持たない（持つと陳腐化し、中核原則に反する）。
- **ガイドに無い観点は適用しない。** ガイドが版・モデルごとに異なる指針を示していれば、対象
  モデルの記述を優先する。

### Step 5: 最適化版と根拠を提示する（出力フォーマット参照）

書き換えた新プロンプトと、**各変更がどのガイド記述に基づくか**を提示する。根拠を示すことで、
ユーザーが判断でき、ガイド更新時に追随できる。

### Step 6:（任意）実測検証へ委譲する

文面上の準拠は「効く」ことの保証ではない。頻用・中核のプロンプトで実際の効果を測りたい場合は、
`empirical-prompt-tuning` skill に引き継ぐ（この skill 内で実測ループは再実装しない）。
そのハンドオフを最後に案内する。

## 出力フォーマット

必ず次の構成で提示する:

```
## 参照した公式ドキュメント
- <取得した URL / モデル別ページ>（対象モデル系: <family>）

## 適用した原則と根拠
- <変更点>: <取得ガイドのどの記述に基づくか>
- ...

## 最適化後プロンプト
<コピペで使えるコードブロック 1 つ。これ単体で成立させる>

## 未解決・要判断点
- <推測で埋めず残した曖昧点。無ければ「なし」>

## 実測検証（任意）
- 効果を測る場合は empirical-prompt-tuning へ。<1 行案内>
```

## 環境制約・フォールバック

- **ドキュメント取得不能時**（ネットワーク無し等）: 最適化を強行しない。「公式ドキュメントを
  取得できなかった」と明示し、(a) ユーザーに URL 提供や再試行を依頼するか、(b) やむを得ず記憶に
  基づく暫定案を出す場合は**「陳腐化している可能性あり・要検証」と必ずラベルする**。黙って記憶で
  埋めるのは NG（この skill の中核原則に反する）。
- **対象モデル系が `doc-sources.md` に無い**: `doc-sources.md` の discovery 手順（公式ドメインを
  web 検索）で正典を探す。見つからなければユーザーに確認。

## Red flags（合理化に注意）

| 出てくる合理化 | 実態 |
|---|---|
| 「ベストプラクティスは知っているから取得不要」 | 訓練データは古い。世代交代で定石は変わる。必ずライブ取得する。 |
| 「対象モデルは自分と同じだろう」 | ユーザーが別モデル向け・複数 family 向けに書いていることは多い。明示指定を最優先し、無ければ family を判定する。 |
| 「意図はだいたい分かるから最適化を進める」 | 目的の誤読は最適化を無意味にする。曖昧なら先に確認する。 |
| 「取得ガイドに無い定石も足しておこう」 | この skill にチェックリストを持ち込むと陳腐化する。適用観点は取得ガイドから抽出する。 |
| 「根拠は省いて最適化版だけ出す」 | 根拠が無いと判断もガイド追随もできない。変更ごとに出典を紐付ける。 |

## 関連スキル

- `empirical-prompt-tuning` — subagent で実測して反復改善する方法論（curated で常在）。本 skill が
  「公式ガイド準拠へ**書き換える**」のに対し、あちらは「書き換えたものが実際に**効くか測る**」。Step 6 の委譲先。
- `skill-creator` — skill 自体の新規作成・改訂（external 取得のため未展開のこともある）。展開されて
  いれば、作成した SKILL.md のプロンプトを磨く段で本 skill を併用できる、程度の補足。
