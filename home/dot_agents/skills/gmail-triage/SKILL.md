# gmail-triage

「201 件の未読」を「読むべき 5 件」に圧縮する。Gmail MCP（`mcp__claude_ai_Gmail__*`）が
利用できない場合は、その旨を伝えて中断する（代替手段で無理に続行しない）。

## 引数

| 引数 | 既定 | 意味 |
|------|------|------|
| `--days` | 14 | スキャン対象期間（`newer_than:<N>d`） |
| `--apply` | off | 分類結果に基づき Gmail ラベルを付与する（付与前に承認必須） |
| `--receipts` | off | 経費・領収書メールのみを対象に、経費精算用の一覧（日付・発行元・金額・件名）を出す |

## 安全原則

- 既定では **メールボックスへの書き込みを一切しない**（`search_threads` / `get_thread` のみ）。
- `--apply` 時もラベル付与のみ。**アーカイブ・削除・既読化・送信は本 skill では行わない**
  （現行 Gmail MCP にアーカイブ操作がないため。恒久的な自動仕分けは Gmail フィルタ側で行う）。
- メール本文の引用は必要最小限にとどめ、認証コード・トークン類はレポートに転記しない。

## Phase 1: 収集

`search_threads` で対象を取得する（`pageSize=50`、`nextPageToken` があれば追加取得。
5 ページを超える場合は打ち切り、溢れた旨をレポート末尾に明記する）:

```
query: "is:unread in:inbox newer_than:<N>d"
```

`--receipts` 時は未読に限定せず、期間内の経費候補を直接検索する:

```
query: "in:inbox newer_than:<N>d {領収書 レシート receipt invoice 請求}"
```

## Phase 2: 分類

送信者ドメイン・件名・snippet から以下の 6 区分に分類する。判定に迷うものは
上位の区分（より人間の目に触れる側）へ倒す:

| 区分 | 判定基準の例 | レポートでの扱い |
|------|-------------|----------------|
| A: 要アクション | 個人アドレスからの直接メール、返信待ち、契約・請求の依頼、クライアント関連ドメイン | **全件列挙**（最優先） |
| B: 経費・領収書 | Stripe / Apple / Google Payments、件名に領収書・receipt・invoice | **全件列挙**（金額があれば併記） |
| C: セキュリティ・アカウント | accounts.google.com、Search Console、ログイン通知、権限変更 | **全件列挙**（異常の可能性を一言評価） |
| D: イベント | connpass、Slack ワークスペース通知、コミュニティ運営連絡 | 件数 + 締切・開催が近いものだけ抜粋 |
| E: 営業・スカウト | LinkedIn、LAPRAS、スカウト媒体、営業 DM | 件数のみ |
| F: ニュースレター・マーケ | プロダクト newsletter、キャンペーン、プロモーション | 件数のみ |

## Phase 3: レポート出力

```markdown
# Gmail Triage — <日付 JST>（対象: 直近 <N> 日 / 未読 <総数> 件）

## A: 要アクション <n> 件
| 日付 | 差出人 | 件名 | 要旨・推奨対応 |

## B: 経費・領収書 <n> 件
| 日付 | 発行元 | 件名 | 金額（判明分） |

## C: セキュリティ・アカウント <n> 件
| 日付 | 発行元 | 件名 | 所見 |

## D〜F: 低優先 <n> 件（イベント <d> / 営業・スカウト <e> / ニュースレター <f>）
（締切が近いイベントのみ抜粋）

## 推奨ネクストアクション
1. <最もレバレッジの高い 1 手>
```

**推奨ネクストアクション**では、該当があれば具体的なハンドオフを提案する:

- B が 3 件以上 → 「`gmail-triage --receipts` で経費一覧を一括回収できます」
- E・F の同一差出人が繰り返し出現 → Gmail フィルタ（自動アーカイブ/ラベル）の作成条件を提示する
- A にクライアント・会社関連の返信待ちがある → 返信ドラフトの作成を提案する（送信はしない）

## Phase 4: ラベル付与（`--apply` 時のみ）

1. `list_labels` で既存ラベルを確認し、不足があれば `AI/要アクション` `AI/経費` `AI/セキュリティ`
   `AI/イベント` `AI/営業` `AI/ニュースレター` を `create_label` で作成する。
2. **付与計画（区分ごとの件数と対象）を提示し、user 承認を得てから** `label_thread` で付与する。
3. 付与後、区分ごとの付与件数を報告する。失敗したスレッドは silent skip せず列挙する。

## 運用メモ

- 恒久的なノイズ削減はラベルではなく Gmail フィルタが本筋。E・F で頻出する差出人は、
  レポート内で「フィルタ化推奨リスト」として `from:` 条件を列挙する（フィルタ作成自体は
  MCP 非対応のため user が Gmail 設定で行う）。
- morning-brief から呼ばれる場合は report-only（Phase 1〜3）で完結させる。
